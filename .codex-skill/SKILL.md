# Task Skill — AI 干活规范

## 核心原则

**先想清楚，再动手。修改前先确认，不乱动无关文件。**

---

## 第一步：理解任务（动手前必做）

收到任务后，先在脑子里过一遍：

1. **目标是什么？** 用一句话复述任务目标，确认理解正确
2. **影响范围？** 哪些文件/模块会被改动，哪些不会
3. **有歧义吗？** 如果有不确定的地方，先提问再动手，不要自己猜

> 如果任务描述清晰，直接输出执行计划；如果有歧义，先问用户 1-2 个关键问题。

---

## 第二步：输出执行计划

动手前列出计划，格式如下：

```
## 执行计划

**目标**：xxx

**改动文件**：
- src/xxx/Foo.java — 新增 xxx 方法
- src/xxx/Bar.java — 修改 yyy 逻辑

**不涉及的文件**：（如果用户提到了担心误改的文件，在这里列出）

**执行步骤**：
1. 先做 A
2. 再做 B
3. 最后做 C
```

**等用户确认后再执行**，除非用户明确说"直接做"。

---

## 第三步：执行规范

### 代码风格

- **命名**：类名 PascalCase，方法/变量 camelCase，常量 UPPER_SNAKE_CASE
- **注释**：
    - 公共方法必须有 Javadoc（或对应语言的文档注释）
    - 复杂逻辑写行内注释，说明"为什么"而不是"是什么"
    - 不写废话注释（`// 给 i 加 1` 这种）
- **方法长度**：单个方法超过 80 行考虑拆分

### 修改纪律

- **最小改动原则**：只改任务涉及的代码，不顺手"优化"无关部分
- **不删用户代码**：除非任务明确要求删除，否则保留原有逻辑（可以注释掉但不直接删）
- **新增 vs 修改分开说**：在输出中明确区分哪些是新增、哪些是修改

### 输出格式

每个改动文件单独给出，带文件路径标题：

```
### src/main/java/com/example/FooService.java

```java
// 代码内容
```
```

---

## 第四步：完成后自检

代码写完后，自己过一遍：

- [ ] 是否实现了任务要求的全部功能？
- [ ] 有没有动到计划外的文件？
- [ ] 命名和注释是否符合规范？
- [ ] 有没有遗留的 TODO / 硬编码 / 临时代码没说明？
- [ ] 与预期结果一致？
自检通过后，在回复末尾附上：

```
## 完成摘要

- 改动文件：xxx
- 新增：xxx
- 修改：xxx
- 注意事项：（如有）
```

## 快速参考

| 情况 | 做法 |
|------|------|
| 任务清晰 | 列计划 → 等确认 → 执行 → 自检 |
| 任务有歧义 | 先问 1—n个问题，直到需求清楚，再列计划 |
| 涉及大范围改动 | 拆成多个小步骤，分批确认 |
| 用户说"直接做" | 跳过确认环节，直接执行并附摘要 |
